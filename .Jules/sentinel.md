## 2026-04-06 - Fix TOCTOU vulnerability in search config
**Vulnerability:** Time-Of-Check to Time-Of-Use (TOCTOU) race conditions in `src/search/config.rs` when loading and saving configuration files. Using `.exists()` before `fs::read_to_string` or `fs::create_dir_all` allowed the file system state to change between the check and the use.
**Learning:** Checking if a file or directory exists before acting on it is a common pattern that introduces TOCTOU vulnerabilities. For example, if a file is deleted between the `exists()` check and the `read_to_string` call, the read will fail.
**Prevention:** Unconditionally perform the desired file system operation (`fs::read_to_string` or `fs::create_dir_all`) and handle the resulting errors directly (e.g., checking for `std::io::ErrorKind::NotFound`). This atomic approach prevents race conditions.
## 2024-05-24 - Secure File Permissions for Configurations
**Vulnerability:** Configuration files and registries potentially containing sensitive data (like `mcp.toml`, `webchatproxy.toml`, and provider config files) were being written using `fs::write()` which creates files with default, potentially overly-permissive system permissions (often 0o644).
**Learning:** Even if the main `keys.toml` file was correctly secured using `OpenOptions`, other configuration files that might inadvertently or explicitly contain sensitive parameters (like authentication tokens or API keys passed to daemon registries) were left exposed.
**Prevention:** Always use `std::fs::OpenOptions` (or `tokio::fs::OpenOptions` for async) with `std::os::unix::fs::OpenOptionsExt` to explicitly set file permissions to `0o600` (read/write for owner only) before creating/opening the file, and verify with `set_permissions` for existing files. This prevents unauthorized users on the same system from reading sensitive configuration data.
## 2024-05-24 - Path Traversal in Sync Config Files
**Vulnerability:** A critical path traversal vulnerability was found in the configuration syncing process. The `handle_sync_from` function used `PathBuf::join` directly with externally-provided configuration filenames (e.g., from an S3 bucket) without any validation. An attacker could craft a sync file with a name like `../../etc/passwd` to overwrite arbitrary files on the local filesystem during a sync operation.
**Learning:** In Rust, `PathBuf::join` completely replaces the base path if the appended string is an absolute path, and it naturally traverses upwards if the string contains `..`. We must never trust filenames derived from remote sources or user input when joining them to local base directories.
**Prevention:** Always validate untrusted/external filenames by explicitly checking `!name.contains("..") && !name.starts_with('/') && !name.starts_with('\\')` before joining them to a base directory to prevent path traversal and arbitrary file write vulnerabilities.
